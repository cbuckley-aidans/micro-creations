Modelled example

Thinking like a programmer

Breaking a problem into jobs.

I'll show you exactly how I work this out, one step at a time.

1

The problem

At the school fete, you are running the lemonade stall. You want the micro:bit to count each cup you sell. When you press button A, it should show how much money you have made. Each cup costs $2.

Before I write any code, I read the problem twice.

Then I look for the doing words.

Each doing word is a clue about what my program has to do.

2

Step 1 · Find the events

What makes something happen?

An event is a trigger. Something that happens that the program reacts to.

EventA cup is sold.
EventButton A is pressed.

Two triggers, so two events.

3

Step 2 · Find the actions

What should the program do?

When a cup is soldAdd one to the count.
When button A is pressedWork out the money. Cups times $2.
ThenShow the money on the screen.
4

Step 3 · Pair them into jobs

A job is an event joined to an action.

Whena cup is sold
→
do thisadd one to the count
Whenbutton A is pressed
→
do thiswork out the money
Thenstraight after
→
do thisshow the money
5

The jobs

  1. Keep a count of how many cups have been sold.
  2. Each time a cup is sold, add one to the count.
  3. When button A is pressed, work out the money (cups × $2).
  4. Show the money on the screen.

That is the whole program, planned before I wrote a single line of code.

6

One more question

What does the program need to remember?

The number of cups. The program has to hold onto that number and change it as cups are sold.

Anything the program remembers or works out is a variable.

{ }

From jobs to code

Once the jobs are clear, the code almost writes itself.

from microbit import * cups = 0 while True: if button_b.was_pressed(): cups = cups + 1 if button_a.was_pressed(): money = cups * 2 display.scroll(money)

Each job became a few lines. Don't worry about the exact words yet, just notice the shape.
